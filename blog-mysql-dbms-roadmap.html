Entity-Relationship Modeling
ER Concepts
  • Entities (strong, weak)
  • Attributes: simple, composite, multivalued, derived
  • Relationships: binary, ternary, n-ary
  • Cardinality: 1:1, 1:N, M:N
  • Participation: partial vs total
  • ISA (inheritance) relationships
  • Chen notation vs Crow's Foot notation
ER → Relational Mapping
  • Mapping strong entities
  • Mapping weak entities
  • Mapping 1:1 relationships
  • Mapping 1:N relationships
  • Mapping M:N (junction/bridge table)

  • Mapping ISA hierarchies
Normalization (all normal forms)
Functional Dependencies
  • What is a functional dependency (FD)
  • Trivial vs non-trivial FDs
  • Armstrong's axioms: Reflexivity, Augmentation, Transitivity
  • Closure of attributes
  • Minimal cover (canonical cover)
  • Finding candidate keys from FDs
1NF, 2NF, 3NF
  • 1NF: atomic values, no repeating groups
  • 2NF: no partial dependency (on composite key)
  • 3NF: no transitive dependency
  • Decomposing to 1NF, 2NF, 3NF with examples
  • Lossless-join decomposition
  • Dependency preservation
BCNF, 4NF, 5NF, 6NF
  • BCNF (Boyce-Codd): every determinant is a superkey
  • BCNF vs 3NF: when they differ
  • 4NF: no multi-valued dependencies (MVD)
  • 5NF (PJNF): join dependencies
  • 6NF: temporal databases
  • DKNF (Domain-Key Normal Form)
Denormalization

  • Data redundancy tradeoffs
  • Denormalization patterns: duplicated columns, pre-aggregated tables, flattened hierarchies
  • OLTP (normalized) vs OLAP (denormalized)
FAANG tip: In system design interviews, you'll be asked to design schemas for products like Twitter, Uber, or Instagram. Knowing 3NF vs denormalization tradeoffs is essential for justifying your choices.
Phase 5 Transactions, Concurrency & ACID
ACID properties · Isolation levels · Locking · MVCC
▼
ACID & Transactions
ACID Properties
  • Atomicity — all or nothing
  • Consistency — valid state to valid state
  • Isolation — concurrent execution
  • Durability — committed = permanent
  • BASE vs ACID (NoSQL tradeoff)
  • CAP theorem: Consistency, Availability, Partition tolerance
Concurrency Problems
  • Dirty read
  • Non-repeatable read
  • Phantom read
  • Lost update
  • Write skew
  • Read skew
Isolation Levels
  • READ UNCOMMITTED — allows dirty reads
  • READ COMMITTED — no dirty reads
  • REPEATABLE READ — MySQL default (InnoDB)
  • SERIALIZABLE — strictest, slowest
  • Which problems each level prevents
  • SET TRANSACTION ISOLATION LEVEL
  • @@transaction_isolation system variable
Locking & MVCC
Lock Types
  • Shared (S) lock — read lock
  • Exclusive (X) lock — write lock
  • Intention locks: IS, IX, SIX
  • Row-level locks vs table-level locks
  • Gap locks, Next-key locks, Record locks
  • Page-level locking
  • SELECT ... FOR UPDATE (X lock)
  • SELECT ... LOCK IN SHARE MODE (S lock)
  • LOCK TABLES / UNLOCK TABLES
Deadlocks
  • What is a deadlock
  • Deadlock detection in InnoDB
  • SHOW ENGINE INNODB STATUS
  • innodb_deadlock_detect variable
  • Deadlock prevention strategies
  • Wait-for graph
  • Lock timeout: innodb_lock_wait_timeout
MVCC
  • Multi-Version Concurrency Control concept
  • Undo log — old row versions
  • Read view — snapshot at transaction start
  • Consistent non-locking reads
  • How REPEATABLE READ uses MVCC
  • Purge thread — old version cleanup
Schedule Theory
  • Serial vs Serializable schedules
  • Conflict serializability
  • Precedence graph (test for conflict serializability)
  • View serializability
  • Recoverability, Cascadeless, Strict schedules
  • Two-Phase Locking (2PL): growing + shrinking
  • Strict 2PL
FAANG tip: Isolation levels and deadlocks are hot topics in backend/infra interviews. Be ready to explain MVCC in InnoDB, why REPEATABLE READ doesn't prevent phantom reads with SELECT, and how gap locks work.
Phase 6 Indexing, Query Optimization & Storage
B+ Tree · Query execution · EXPLAIN · InnoDB internals
▼
Indexes — Deep Dive
Index Types
  • B-Tree index (default in InnoDB)
  • Hash index (MEMORY engine)
  • Full-text index (MATCH ... AGAINST)
  • Spatial index (R-Tree, for geometry)
  • Primary index (clustered)
  • Secondary index (non-clustered)
  • Unique index
  • Composite (multi-column) index
  • Covering index
  • Prefix index
  • Invisible index (MySQL 8+)
  • Functional/Expression index (MySQL 8.0.13+)
B+ Tree Internals
  • Tree structure: root, internal nodes, leaf nodes
  • All data at leaf level
  • Doubly linked leaf nodes (range scans)
  • Page size (default 16KB)
  • Fill factor and page splits
  • Height and its impact on lookups
  • Clustered index: data IS the index
  • Secondary index: stores primary key value
  • Double lookup for secondary indexes
Index Design Rules
  • Selectivity: high cardinality = better index
  • Left-prefix rule for composite indexes
  • Index merge optimization
  • When MySQL doesn't use an index
  • Index bloat from writes
  • Too many indexes = slow writes
  • Index on foreign keys
  • FORCE INDEX, USE INDEX, IGNORE INDEX hints
  • Covering index: avoids table access
  • Index condition pushdown (ICP)
Query Optimization
EXPLAIN & EXPLAIN ANALYZE
  • EXPLAIN columns: id, select_type, type, key, rows, Extra
  • Access types: system, const, eq_ref, ref, range, index, ALL
  • select_type: SIMPLE, PRIMARY, SUBQUERY, DERIVED, UNION
  • Extra values: Using index, Using filesort, Using temporary
  • EXPLAIN FORMAT=JSON
  • EXPLAIN ANALYZE (MySQL 8.0.18+, actual timing)
Query Execution
  • Query parsing → validation → optimization → execution
  • Cost-based optimizer (CBO)
  • Join order optimization
  • Statistics: ANALYZE TABLE, information_schema.INNODB_TABLE_STATS
  • Optimizer hints (MySQL 8+): INDEX_MERGE, JOIN_ORDER, etc.
  • Query cache (removed in MySQL 8)
  • Subquery flattening / materialization
  • Derived condition pushdown
Performance Schema & Profiling
  • SHOW PROFILES / SHOW PROFILE
  • Performance schema tables: events_statements_summary
  • sys schema views: sys.statements_with_full_table_scans
  • Slow query log: slow_query_log, long_query_time
  • pt-query-digest (Percona tool)
  • INFORMATION_SCHEMA queries
Storage Engine Internals (InnoDB)
InnoDB Architecture
  • Buffer pool: pages, LRU list, dirty pages
  • Change buffer (secondary index buffering)
  • Adaptive hash index (AHI)
  • Log buffer → redo log files
  • Undo tablespace
  • Doublewrite buffer
  • Page: 16KB, header + data + trailer
InnoDB Files & Config
  • ibdata1: system tablespace
  • ibd files: per-table tablespace (innodb_file_per_table)
  • ib_logfile0/1: redo log
  • innodb_buffer_pool_size (most critical)
  • innodb_flush_log_at_trx_commit (0,1,2)
  • innodb_flush_method
  • Other storage engines: MyISAM, MEMORY, ARCHIVE, CSV
WAL & Crash Recovery
  • Write-ahead logging (WAL) concept
  • Redo log: how writes are guaranteed durable
  • Undo log: rollback and MVCC
  • Checkpoint: LSN (log sequence number)
  • Crash recovery: redo then undo
  • Fuzzy checkpoint
FAANG tip: "How would you debug a slow query?" is an almost universal interview question. Walk through: EXPLAIN, check access type (avoid ALL/index), look for Using filesort, check index selectivity, consider covering indexes.
