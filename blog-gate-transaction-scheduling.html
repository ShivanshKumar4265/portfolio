GATE: Transaction Scheduling — Concepts & Practice

A comprehensive guide to tracing database schedules, identifying anomalies, understanding serializability, and solving GATE PYQs.

Back to Portfolio

CORE CONCEPT: How to Trace a Schedule

A schedule is an interleaved sequence of operations from multiple transactions. To find final values, you simulate execution step by step, tracking what each variable holds in memory (local variable) vs. what's written to the database.

Key Mental Model

Database (disk) ──Read──▶ Transaction's local variable ──compute──▶ Write ──▶ Database (disk)
  • Read(X) → copies DB value of X into transaction's local copy
  • Write(X) → pushes transaction's local copy of X back to DB
  • Any computation (A = A - 10) only affects the local copy until Write is called

Step-by-Step: Solving the Given Question

Initial: A = 100, B = 200

2T1A = A−10A=90—100200
3T2Read(A)A=90A=100100200
4T2Temp = 0.2×A—Temp=20100200
5T2Write(A)—A=100,T=20100200
6T2Read(B)—B=200100200
7T1Write(A)A=90—90200
8T1Read(B)B=200—90200
9T1B = B+10B=210—90200

Final: A = 90, B = 220 → A + B = 310

Notice: T2 Read(B) at step 6 captured B=200 (before T1's write at step 10). So T2 computes B = 200 + 20 = 220, overwriting T1's write. This is a Lost Update anomaly.

Critical Anomalies You Must Recognize

1. Lost Update

T1 and T2 both read X, both modify it, T2 writes last → T1's update is lost.

T1: Read(X)        X=100
T2: Read(X)        X=100
T1: X=X+10, Write(X)   DB: X=110
T2: X=X+20, Write(X)   DB: X=120  ← T1's +10 is LOST

2. Dirty Read (WR conflict)

T1 writes X, T2 reads X, T1 rolls back → T2 read a never-committed value.

3. Unrepeatable Read (RW conflict)

T2 reads X, T1 writes X, T2 reads X again → different values in same transaction.

4. Phantom Read

T2 reads a set of rows, T1 inserts a new row, T2 reads again → extra row appears.

Serializability — The Key Theory

Conflict Serializability (most tested in GATE)

Two operations conflict if:

  • They belong to different transactions
  • They access the same data item
  • At least one is a Write

Conflict pairs: RW, WR, WW (NOT RR)

Precedence Graph (Serialization Graph)

  • Draw node for each transaction
  • For each conflict, draw edge: Ti → Tj if Ti's operation comes BEFORE Tj's conflicting operation
  • No cycle → Conflict Serializable
  • Cycle exists → NOT conflict serializable

View Serializability

Harder to test, rarely asked computationally. Just know:

  • Every conflict-serializable schedule is view-serializable
  • Converse is NOT true (blind writes can make it view-serializable but not conflict-serializable)

Recoverability

Type Condition GATE Frequency
RecoverableIf Ti reads from Tj, Tj must commit before Ti commitsHigh
Cascadeless (ACA)Ti can only read committed dataHigh
StrictTi cannot read OR write X until the last transaction to write X commitsMedium

Hierarchy: Strict ⊂ Cascadeless ⊂ Recoverable ⊂ All Schedules

Lock-Based Protocol Quick Reference

Protocol Rule Guarantees
2PLNo new lock after any unlockConflict Serializability
Strict 2PLHold all X-locks till commitStrict schedule
Rigorous 2PLHold ALL locks till commitStrict + Cascadeless
Conservative 2PLAcquire all locks before startNo Deadlock

2PL does NOT prevent deadlock (except Conservative 2PL).

Section B — Conflict Serializability

Q4 (Medium): Is S conflict serializable?

Schedule S: R1(A), R2(B), W1(A), R1(B), W2(B), W1(B)

Answer:

Conflicts:

  • W1(A) vs ? — no T2 operation on A → no conflict on A
  • R1(B) → W2(B): T1 reads B before T2 writes → T1 → T2
  • W2(B) → W1(B): T2 writes B before T1 writes → T2 → T1
  • R1(B) → W1(B): same transaction, skip

Precedence graph: T1 → T2 AND T2 → T1 → CYCLE!
Not conflict serializable.

Q5 (Hard — GATE 2019 pattern): Edges in precedence graph

Schedule S: R1(X), R2(X), R3(Y), W1(X), R2(Y), W2(X), W2(Y), W3(Y)

How many edges does the precedence graph have?

Answer:

Conflicts (different transactions, same item, at least one write):

On X:

  • R1(X) → W2(X): T1→T2
  • R2(X) → W1(X): T2→T1
  • W1(X) → W2(X): T1→T2 (already counted, but same edge)
  • R2(X) → W2(X): same transaction, skip

On Y:

  • R3(Y) → W2(Y): T3→T2
  • R3(Y) → W3(Y): same transaction, skip
  • R2(Y) → W3(Y): T2→T3
  • W2(Y) → W3(Y): T2→T3 (same edge)

Unique edges: T1→T2, T2→T1, T3→T2, T2→T3

4 edges. Cycle exists (T1↔T2 and T2↔T3). Not conflict serializable.

Section C — Recoverability

Q6 (Medium): Which is true? 
T1: R(A), W(A), ........Abort
T2: ........R(A), W(A), Commit

(T2 reads A after T1 writes A, but before T1 aborts)

(A) Schedule is recoverable and cascadeless
(B) Schedule is recoverable but NOT cascadeless
(C) Schedule is NOT recoverable
(D) Schedule is strict

Answer:

T2 reads A written by T1 (dirty read). T1 aborts AFTER T2 commits.

  • T2 committed before T1 aborted → T2 committed based on dirty data → NOT recoverable

Answer: (C)

Q7 (Tricky): Is this recoverable? Cascadeless? 
T1: R(X), W(X)
T2: R(X), W(X), Commit
T1: Commit

(T2 reads X after T1 writes X)

Answer:

  • T2 reads X written by T1 (dirty read at time of read — T1 not yet committed)
  • T2 commits BEFORE T1 commits
  • If T1 aborts after T2 commits → problem. But here T1 commits (successfully).
  • Still, at the time T2 read, T1 was uncommitted → violates cascadeless (ACA)
  • But T1 ultimately commits before... wait, T2 commits BEFORE T1.
  • For recoverability: T2 read from T1, so T1 must commit before T2. Here T2 commits first → NOT recoverable.

Answer: Not recoverable (and hence not cascadeless or strict)

Section D — 2PL and Locking

Q8 (Medium): 2PL anomaly

A schedule follows 2PL. Which anomaly can still occur?

(A) Dirty Read
(B) Lost Update
(C) Deadlock
(D) Non-serializable execution

Answer:

2PL guarantees conflict serializability, preventing dirty reads, lost updates, and non-serializable execution. However, 2PL does NOT prevent deadlock.

Answer: (C) Deadlock

Q9 (Hard — GATE 2022 style): 2PL & Deadlock 
T1: Lock-X(A), R(A), W(A), Lock-X(B), Unlock(A), R(B), W(B), Unlock(B)
T2: Lock-X(B), R(B), W(B), Lock-X(A), Unlock(B), R(A), W(A), Unlock(A)

Does this follow 2PL? Can deadlock occur?

Answer:

2PL check:

  • T1: Acquires Lock-X(A), then Lock-X(B), then starts unlocking → Growing phase ends at Lock-X(B). Unlock(A) comes after Lock-X(B). ✓ 2PL satisfied for T1.
  • T2: Similar pattern. ✓ 2PL satisfied for T2.

Deadlock check:

  • If T1 holds Lock-X(A) and waits for Lock-X(B)
  • AND T2 holds Lock-X(B) and waits for Lock-X(A)
  • → Deadlock! (Circular wait)

Answer: Yes, follows 2PL. Yes, deadlock can occur (classic deadlock scenario).

Section E — Timestamp-Based & MCQ Traps

Q10 (Hard — Conceptual Trap): View Serializability

Consider a schedule that is view serializable but NOT conflict serializable. What is the minimum number of blind writes required to make this possible?

Answer:

At least 1 blind write (a write not preceded by a read of the same data item by the same transaction).

Example:

T1: W(X)
T2: R(X), W(X)
T3: W(X)

Schedule: W1(X), W3(X), R2(X), W2(X)

T3 writes X blindly. Conflict graph has cycle, but it's view-equivalent to serial schedule T1,T3,T2.

Answer: 1

Quick Formula Sheet

Lost Update           → WW or RW interleaving without proper locking
Dirty Read            → W then R before commit, then abort
Unrepeatable Read     → R...W...R (by different transactions) on same item
Conflict Serializable → No cycle in precedence graph
2PL                   → All locks before any unlock (growing/shrinking phases)
Recoverability        → Ti reads from Tj → Tj commits before Ti
Cascadeless           → Only read committed values
Strict                → Don't read OR write X until writer commits
