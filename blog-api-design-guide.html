01

What is an API?

An API (Application Programming Interface) acts as a formalized, structured contract between two software boundaries. It establishes exactly what resources and endpoints are exposed, what arguments or payloads must be supplied, and what payload structure is guaranteed to return. The consuming boundary interacts purely through this contract, completely isolated from internal database configurations, state storage mechanics, or implementation language specifications.

In modern systems architecture, selecting the appropriate API paradigm is a critical architectural decision based on data shape, bandwidth requirements, and streaming characteristics:

API Type Core Protocol Data Model & Mechanics Primary Use Case
REST HTTP/1.1 or HTTP/2 Resource-centric representation, structured JSON/XML payloads, standard CRUD commands mapped to HTTP methods. Public web APIs, enterprise platform integrations, standard CRUD client-server products.
GraphQL HTTP/1.1 or HTTP/2 Graph model queries. Single POST gateway endpoint where clients submit specific schema selections. Single Page Applications (SPAs), mobile clients where network bandwidth and round-trip times are premium.
gRPC HTTP/2 only High-performance RPC using binary Protocol Buffers (Protobuf). Strongly typed schemas compiled into client/server stubs. High-speed microservices inter-communication, low-overhead cloud integrations.
SOAP HTTP, SMTP, TCP XML-wrapped message exchanges requiring WSDL definitions. Rigid, legacy enterprise standard. Legacy banking platforms, highly audited transactional legacy services.
WebSocket TCP / WS Upgrade Persistent, full-duplex TCP channels allowing bi-directional message frames without polling overhead. Real-time chat platforms, collaborative document editing, live gaming feeds.

Architectural Context

While REST remains the default choice for external public integrations due to its high cacheability and simple tooling, gRPC has largely taken over internal service-to-service communication paths in modern microservice architectures because it dramatically reduces JSON encoding/decoding overhead and enforces strict compile-time interface schemas.

02

REST API Fundamentals

REST (Representational State Transfer) is an architectural style designed around Resources rather than operations. A resource is an entity identified by a uniform URI (e.g. `/users/42`) and manipulated using standard HTTP verbs.

HTTP Methods (CRUD Mapping)

To build high-fidelity REST APIs, HTTP methods must map correctly to CRUD database logic, adhering strictly to the specifications of safety (not modifying state) and idempotency (repeated executions returning identical side effects):

HTTP Method CRUD Action Idempotent? Safe? Example Target
GET Read (Fetch resource) Yes Yes `GET /users/42` — Fetches profile of User 42.
POST Create (Instantiate resource) No No `POST /users` — Registers a new user. Generates new ID.
PUT Full Update (Replace target) Yes No `PUT /users/42` — Completely replaces User 42 schema.
PATCH Partial Update (Modify subset) No (Usually) No `PATCH /users/42` — Modifies specific fields of User 42 profile.
DELETE Delete (Remove resource) Yes No `DELETE /users/42` — Removes User 42 permanently.

HTTP Status Codes

Status codes inform consuming clients of the exact outcome of their request. Standardizing on expressive status codes prevents developers from nesting custom error formats inside generic `200 OK` envelopes:

Range Category Critical Codes & Usage
2xx Success 200 OK: Request succeeded.
201 Created: POST successfully created a resource.
204 No Content: DELETE/PUT succeeded; no payload body returned.
3xx Redirection 301 Moved Permanently: URI updated.
304 Not Modified: Resource cached; content matches local validation.
4xx Client Error 400 Bad Request: Input validation failed.
401 Unauthorized: Authentication credentials missing or bad.
403 Forbidden: Authenticated, but lacks required permissions (RBAC).
404 Not Found: URI does not map to a resource.
409 Conflict: Resource state conflict (e.g., email already registered).
429 Too Many Requests: Rate limit threshold breached.
5xx Server Error 500 Internal Error: Uncaught exception or database crash.
503 Service Unavailable: Node overload, maintenance window.

The 6 REST Architectural Constraints

  1. Client-Server Separation: Decouples structural presentation boundaries from core persistent engine boundaries, permitting independent horizontal scale.
  2. Statelessness: Every incoming request must package complete operational context, metadata, and security tokens. The server holds zero session states within memory blocks.
  3. Cacheability: Downstream layers (CDN, Proxy, Browser) must receive clear caching metadata (e.g. `Cache-Control` header) defining payload expirations.
  4. Uniform Interface: Uniform URIs, standard HTTP action methods, self-describing resource structures, and hypermedia controls (HATEOAS).
  5. Layered System: The consumer must have zero visibility of system routing paths, unable to determine if it speaks to the source microservice, a load balancer, or an edge CDN proxy.
  6. Code on Demand (Optional): Servers can transmit executable scripts or applets (e.g., client-side JS) to dynamic client boundaries.
03

URL / Endpoint Design

A high-fidelity API must treat URLs as clean, expressive addresses mapping directly to hierarchical resource relationships. Poor URI schemas that treat paths as RPC calls degrade scalability and readability.

HTTP API Schema Guidelines 
# Rule 1: Always utilize plural nouns representing collections. Avoid verbs.
# GOOD: Matches REST standards
GET /orders
GET /orders/123

# BAD: Mixing verbs or RPC style
GET /getOrders
GET /get-order?id=123

# Rule 2: Model hierarchical nested dependencies cleanly. Limit nesting to 2 levels.
# GOOD: Clear resource association
GET /users/42/orders          # Fetch orders belonging to User 42
GET /users/42/orders/789      # Fetch specific order 789 belonging to User 42

# BAD: Too deep, hard to maintain
GET /departments/finance/teams/backend/users/42/projects/12

# Rule 3: Utilize kebab-case rather than camelCase for multi-word collections
# GOOD: Search-engine friendly & easy to read
GET /order-items/55

# BAD: Mix of casing styles
GET /orderItems/55
GET /order_items/55

# Rule 4: Handle actions that don't map to standard CRUD using explicit sub-resources or verbs
POST /payments/123/refund     # Represents a transaction operation
POST /users/42/deactivate     # Changes status via specific workflow

Query Parameters vs Path Parameters

Selecting between query and path parameters is dictated by the conceptual intent of the client's request:

Parameter Type Primary Intent Example Configuration
Path Parameter Locating and identifying a specific resource node within a collection. Necessary for locating single entities. `/products/prod-8871`
Query Parameter Applying modifiers to a collection, such as filtering, sorting, pagination, or optional child inclusion metadata. `/products?category=electronics&sort=price:desc&page=2`

REST API Anti-Pattern

Avoid passing unique resource identifiers within query strings (e.g. `/get-user?userId=42`). This breaks HTTP caching strategies. Proxies and edge CDNs treat unique path nodes differently than query strings, which often get bypassed for static edge caching.

Request & Response Design

API payloads must adhere to consistent schemas, shielding clients from variable output structures and providing descriptive request tracking details.

1. Structured Request Payload

POST /orders 
{
  "userId": "u-123",
  "items": [
    { "productId": "p-456", "quantity": 2 }
  ],
  "shippingAddress": {
    "street": "123 Main St",
    "city": "Bangalore",
    "pincode": "560001"
  }
}

2. Consistent Response Structure

Never return raw data arrays at the root level of a JSON payload. Always wrap payloads in a root object containing metadata, facilitating forward-compatible structural upgrades without breaking arrays.

JSON Response Standard 
{
  "success": true,
  "data": {
    "orderId": "o-789",
    "status": "confirmed",
    "total": 1500.00
  },
  "meta": {
    "requestId": "req-abc-123",
    "timestamp": "2026-05-25T10:30:00Z"
  }
}

3. Standardized Error Response

When errors occur, return a highly descriptive payload highlighting validation problems, while sanitizing stack traces to protect system internals.

JSON Error Standard 
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid input data",
    "details": [
      { "field": "email", "issue": "Must be a valid email address" },
      { "field": "age", "issue": "Must be at least 18" }
    ]
  },
  "meta": {
    "requestId": "req-abc-123"
  }
}
05

Authentication & Authorization

Securing transactional boundaries requires establishing clear authentication protocols (identifying who the caller is) and authorization policies (governing what operations they are allowed to perform).

Authentication Protocols

  1. API Key: Passed via header e.g., `X-API-Key: secret-key-here`. Primarily optimized for server-to-server microservices. Simple, but lacks expiration.
  2. Basic Auth: Passed via `Authorization: Basic base64(username:password)`. High overhead risk, highly vulnerable to decoding. Avoid in production.
  3. Bearer Token (JWT): Passed via `Authorization: Bearer eyJ...`. Self-contained, cryptographically signed, stateless token.

OAuth 2.0 Authorization Flows

OAuth Flow Primary Target / Client Type Operational Mechanics
Authorization Code with PKCE SPAs, Mobile Apps, Native Platforms Uses code challenge verifiers to exchange authorizations without exposing secret keys in browsers.
Client Credentials Server-to-Server / Background Daemons Machines directly request tokens using their private client IDs and secret key sets. No user context involved.
Authorization Code Traditional server-side web applications Redirects users to auth servers, returns authorization code, backend swaps code for access token.
Implicit Flow Deprecated (Insecure) Tokens returned directly in browser redirect fragment. Highly vulnerable to access-token theft. Avoid.

JWT Structure

A JSON Web Token consists of three distinct parts concatenated by dots: `Header.Payload.Signature`

JWT Format Representation 
# 1. Header: Defines cryptographic signing algorithm
{
  "alg": "HS256",
  "typ": "JWT"
}

# 2. Payload: Packages claims (subject, role, expiration epoch timestamp)
{
  "sub": "user-123",
  "role": "admin",
  "exp": 1748000000
}

# 3. Signature: Created by concatenating Base64 Encoded Header and Payload, then signing with a private server secret key
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  private_secret_key
)

Authorization Strategies

  • RBAC (Role-Based Access Control): Permissions tied to static user roles (e.g. user, manager, admin).
  • ABAC (Attribute-Based Access Control): Dynamic policies evaluating attributes (e.g., location, time window, resource owner).
  • OAuth Scopes: Fine-grained capability access lists (e.g. `read:profile`, `write:billing`) delegating system actions to third-party integrations.
06

Versioning

API Versioning is mandatory to introduce breaking changes (removing fields, re-typing formats, renaming keys) without disrupting existing clients in production.

Versioning Methods 
# 1. URI Versioning (Highly recommended, easily cached by proxies)
GET /v1/users
GET /v2/users

# 2. Header Versioning (Pristine, but complex to query and test manually)
GET /users
Accept: application/vnd.myapi.v2+json

# 3. Query Parameter Versioning
GET /users?version=2

# 4. Subdomain Versioning
GET https://v2.api.example.com/users

Interview Tip

Interviewers favor URI Versioning because it offers maximum clarity, simplifies debugging, and works flawlessly with standard CDN configurations out-of-the-box. When versioning, ensure you deprecate old structures gracefully by passing deprecation timelines using standard HTTP headers (e.g. `Sunset` and `Deprecation`).

Pagination, Filtering & Sorting

Large database tables must never be dumped directly onto network channels. Implementing efficient filtering, sorting, and pagination constraints is essential for performance and reliability.

Pagination Strategies

1. Offset Pagination

Uses simple SQL `LIMIT` and `OFFSET` clauses. Good for small collections, but scales terribly as the database must scan all preceding rows.

GET /products?page=3&limit=20 
{
  "data": [...],
  "pagination": {
    "total": 500,
    "page": 3,
    "limit": 20,
    "totalPages": 25
  }
}

2. Keyset Pagination (Seek Method)

Filters rows using comparative parameters based on high-performance indexed database keys (e.g., `WHERE id > 500 LIMIT 20`). More performant than offsets but harder to implement for complex sort paths.

GET /orders?after_id=500&limit=20 
{
  "data": [...],
  "pagination": {
    "limit": 20,
    "afterId": 500
  }
}

3. Cursor-Based Pagination

Highly optimized, state-isolated technique. The client receives an opaque base64-encoded cursor containing structural metadata. Perfect for highly dynamic or infinite scroll feeds.

GET /feed?cursor=eyJpZCI6MTAwfQ==&limit=20 
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6MTIwfQ==",
    "hasMore": true
  }
}

Filtering & Sorting Specification

Standardize sorting and filtering formats to minimize custom logic in endpoints:

Query Filter Formats 
# Filtering specific fields
GET /products?category=electronics&minPrice=500&maxPrice=5000

# Sorting: Use comma-separated fields with a clear ascending/descending modifier
GET /users?sort=createdAt:desc,name:asc

# Time-series filters
GET /orders?status=pending&createdAfter=2026-01-01
08

Rate Limiting

Rate limiting protects infrastructure from malicious Denial of Service (DoS) attacks, brute-force requests, and resource depletion.

Rate Limiting Algorithms

Algorithm Operational Mechanics Pros / Cons
Fixed Window Limits requests within static time boundaries (e.g., max 100 requests per 1-minute window). Simple to implement. Vulnerable to traffic spikes at window transitions.
Sliding Window Maintains a sorted set of request timestamps to evaluate requests over a rolling window. Very accurate, smooth. Memory intensive due to large set logs.
Token Bucket Tokens added to a bucket at a fixed fill rate. Incoming requests consume a token. Allows bursts when full. Highly flexible. Supports bursts while maintaining long-term rate constraints.
Leaky Bucket Requests flow into a queue and are processed at a steady, fixed rate. Excessive requests overflow and fail. Perfect for smoothing traffic spikes. Adds latency to requests.

Standard Rate Limit Headers

Inform downstream clients of their throttling limits with standard HTTP headers:

HTTP/1.1 429 Too Many Requests 
X-RateLimit-Limit: 1000        # Max allowable allocations within current window
X-RateLimit-Remaining: 0       # Remaining requests available
X-RateLimit-Reset: 1748001600  # Epoch timestamp when the rate limit window resets
Retry-After: 60                # Seconds to wait before retrying

Caching

Caching dramatically improves performance and reduces server load. By configuring caching parameters correctly, we prevent servers from repeatedly generating identical static outputs.

Cache-Control Header Matrix

Cache-Control Directives 
# Cached by proxies, gateways, and clients for 1 hour
Cache-Control: public, max-age=3600

# Strictly cached only by the end-user browser (sensitive data)
Cache-Control: private, max-age=300

# Force clients to validate with origin server before using cached copies
Cache-Control: no-cache

# Strictly ban caching entirely (sensitive financial data)
Cache-Control: no-store

# Strong validation token representing resource fingerprint
ETag: "abc123"

# Weak validation token representing resource modification date
Last-Modified: Mon, 25 May 2026 10:00:00 GMT

Multi-Tier Caching Architecture

Client → Edge CDN → API Gateway Proxy → Redis Cache → Database

HTTP Conditional Requests

Avoid sending full payloads when resources haven't changed. Clients can pass resource tags back to the server in validation requests:

Validation Handshake 
# 1. Client conditional request
GET /products/42
If-None-Match: "abc123"

# 2. Server response (If resource matches the client tag, skip body serialization)
HTTP/1.1 304 Not Modified
Cache-Control: public, max-age=3600
ETag: "abc123"

Idempotency

An operation is idempotent if executing it multiple times produces the exact same resource state as executing it a single time. Idempotency is crucial for handling network retries safely, preventing duplicate transactions (like double-billing a user).

HTTP Method Idempotency Matrix

HTTP Method Safe? Idempotent? Operational Side Effect Details
GET Yes Yes Pure reading operations. Never modifies state.
PUT No Yes Repeated PUT operations replace the target with identical data, resulting in the same final state.
DELETE No Yes The first execution deletes the resource. Subsequent executions return `404 Not Found`, but state change is identical.
POST No No Submitting multiple POST requests will create multiple distinct records, leading to duplicates.
PATCH No No (Usually) Can be non-idempotent depending on operation (e.g., appending values or incrementing a field like `{ "price": "$price + 10" }`).

Implementing Idempotent POST Operations

To make non-idempotent endpoints (like payment creation) safe for retries, require clients to pass a unique Idempotency-Key in the header:

POST /payments 
# Request Header
POST /payments
Idempotency-Key: a8098c1a-f86e-11da-bd1a-00112444be1e

{
  "amount": 5000,
  "currency": "INR"
}

Idempotency Execution Flow

  1. The server intercepts the request and checks for an existing `Idempotency-Key` in cache (e.g., Redis).
  2. If the key exists, the server skips execution and returns the cached response immediately.
  3. If the key is new, the server locks the key, processes the request, and caches the final response with a defined TTL.

API Security (OWASP Top 10)

Securing API gateways requires deep familiarity with the OWASP API Security Top 10 vulnerabilities, which target API endpoints specifically.

OWASP Vulnerability Underlying Risk Mitigation Strategy
1. BOLA (Broken Object Level Authorization) Clients access resources simply by modifying IDs in query paths (e.g., accessing `/users/43` from `/users/42`). Enforce strict owner authorization policies on every request. Never rely on unauthenticated path IDs.
2. Broken Authentication Weak token signatures, lack of expiration dates, or predictable API keys. Implement standardized JWT structures with robust cryptographics and strict expirations.
3. Broken Object Property Level Authorization Endpoints expose sensitive internal fields that standard users shouldn't see (e.g., exposing `isAdmin` parameters). Sanitize outgoing payloads. Enforce strict DTO data-mapping contracts.
4. Unrestricted Resource Consumption No rate limiting, letting clients execute massive operations that exhaust server memory, CPU, or bandwidth. Enforce rate limits, pagination maximums, and payload size restrictions.
5. Broken Function Level Authorization Standard users access administrative functions simply by hitting admin routes (e.g. `POST /admin/delete-user`). Implement centralized RBAC filters checking role permissions on every protected path.
6. Unrestricted Access to Sensitive Business Flows Bots exploiting standard workflows (e.g. bulk ticketing or inventory hoarding). Add anomaly detection, client fingerprinting, and device verification constraints to sensitive routes.
7. SSRF (Server-Side Request Forgery) API takes user-supplied URLs and queries them directly, exposing internal network nodes. Sanitize and validate URL inputs. Sandbox external requests using strict DNS/IP blocklists.
8. Security Misconfiguration Exposing verbose debug stack traces in production error outputs. Disable detailed error messages externally. Implement a global exception handler to return clean JSON envelopes.
9. Improper Inventory Management Exposing deprecated or unversioned routes (like `/v1/beta/users`) that bypass security filters. Maintain centralized API schemas. Audit and deprecate old endpoints aggressively.
10. Unsafe Consumption of APIs Blindly trusting payloads returned from third-party APIs without validation. Validate and sanitize all data received from external integrations before processing.

Security Checklist

HTTPS Always

Enforce TLS 1.2+ globally. Reject plain HTTP requests at the gateway.

Input Validation

Validate all incoming query parameters and JSON payloads against a strict schema.

Centralized CORS

Restrict browser access to trusted domains using explicit CORS header configurations.

Token Rotation

Implement short-lived access tokens paired with secure refresh token rotation.

Advanced API Design Patterns

Advanced architectural patterns are key to designing highly scalable, discoverable, and robust systems.

1. HATEOAS (Hypermedia as the Engine of Application State)

Makes APIs self-discoverable by returning related actions and links along with the resource representation:

HATEOAS JSON Response 
{
  "orderId": "o-789",
  "status": "pending",
  "_links": {
    "self": { "href": "/orders/789" },
    "cancel": { "href": "/orders/789/cancel", "method": "POST" },
    "pay": { "href": "/orders/789/pay", "method": "POST" }
  }
}

2. Partial Responses (Field Filtering)

Reduce network payload overhead by letting clients select exactly which JSON fields they want returned:

GET /users/42?fields=id,name,email 
{
  "id": 42,
  "name": "Alex Johnson",
  "email": "alex@api.dev"
}

3. Bulk Operations

Avoid network round-trip overhead by grouping multiple updates or creations into a single array payload:

PATCH /products 
[
  { "id": "p-1", "price": 999 },
  { "id": "p-2", "price": 1499 }
]

4. Long-Running Operations (Async API Flow)

Never block synchronous client connections on heavy backend operations (like video encoding or report generation). Instead, implement a non-blocking asynchronous polling pattern:

Asynchronous Job Pattern 
# Step 1: Client submits job. Server responds immediately with 202 Accepted.
POST /reports/generate
→ HTTP/1.1 202 Accepted
{ 
  "jobId": "job-123", 
  "statusUrl": "/jobs/job-123" 
}

# Step 2: Client polls statusUrl
GET /jobs/job-123
→ HTTP/1.1 200 OK
{ 
  "status": "processing", 
  "progress": 60 
}

# Step 3: Job completes. Server returns final result pointer.
GET /jobs/job-123
→ HTTP/1.1 200 OK
{ 
  "status": "completed", 
  "resultUrl": "/reports/r-456" 
}

API Documentation

An undocumented API is a broken API. Modern teams standardize on the OpenAPI Specification (OAS) to generate interactive documentation portals (like Swagger UI) automatically.

OpenAPI 3.0 (YAML Spec) 
openapi: 3.0.0
info:
  title: Orders API Portal
  version: 1.0.0
paths:
  /orders:
    get:
      summary: Retrieve lists of orders
      parameters:
        - name: status
          in: query
          schema:
            type: string
            enum: [pending, confirmed, delivered]
      responses:
        '200':
          description: Array of orders returned
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Order'

What Good Docs Must Include

  • Clear Auth Guide: Step-by-step instructions on acquiring and passing tokens.
  • Environments Map: Explicit URLs for development, sandbox, staging, and production environments.
  • Expressive Mock Payloads: Working request and response examples for every supported schema.
  • SDK Code Snippets: Multi-language integrations (e.g., cURL, Node.js, Python, Go) showing actual request implementations.

Advanced Topics: Gateway & Scale

Scaling APIs to handle millions of requests requires decoupling infrastructure concerns from core backend services.

1. Centralized API Gateway

An API Gateway acts as a reverse proxy, shielding internal microservices by consolidating infrastructure tasks in a single layer:

  • Routing: Dynamically maps public request paths to internal services.
  • Authentication: Offloads authorization checks from downstream microservices.
  • SSL Termination: Handles public HTTPS handshakes at the edge, communicating with internal nodes via fast, unencrypted paths.
  • Centralized Logging: Aggregates telemetry data for deep system observability.

2. WebSockets vs Server-Sent Events (SSE)

Feature WebSocket SSE (Server-Sent Events)
Direction Bi-directional (Full duplex). Both sides send data. Uni-directional (Server → Client only).
Underlying Protocol TCP / Custom WS Protocol. Standard HTTP.
Re-connection Must be implemented manually in javascript. Handled automatically by the browser.
Best For Real-time chat, multiplayer gaming, collaborative tools. Live stock tickers, notifications, logging streams.

3. Backward Compatibility Rules

To avoid breaking existing client integrations in production, follow these structural rules:

  • DO NOT remove or rename existing JSON fields in response payloads.
  • DO NOT change the data type of existing fields (e.g., changing string IDs to integers).
  • DO add new optional fields or parameters safely.
  • AVOID introducing new mandatory query parameters, as they instantly break older clients.

Interview Cheat Sheet

Key talking points and system design structures to reference during SDE interviews:

"How do you design a highly scalable E-commerce API?"

"I'd establish clean resource hierarchies: `/products`, `/users`, `/orders`. I'd isolate transaction boundaries using token auth, enforce cursor pagination for product feeds to handle scale, implement centralized rate limiting at the API Gateway, and safeguard payments with Idempotency keys."

"How do you prevent double-charging on payment failures?"

"I'd enforce an `Idempotency-Key` header for transactional operations. The server validates this key against Redis before execution. If the key exists, it skips execution and returns the cached result, safely preventing double charges."

"When do you select GraphQL over REST?"

"GraphQL shines when clients need highly variable nested structures, as it eliminates over-fetching and under-fetching. However, REST is superior when high edge cacheability, standardized CDN performance, or simple public integrations are primary requirements."

Golden Rules Summary

Incorporate these ten principles into your engineering practices to build world-class APIs:

1. Design for the Consumer

Treat your API as a product. Prioritize developer ergonomics and predictability.

2. Maintain Strict Consistency

Standardize field naming, casing, pagination parameters, and error formats globally.

3. Be Highly Explicit

Use clear field names and descriptive error messages. Avoid cryptic codes.

4. Fail Gracefully

Return precise status codes paired with validation details to accelerate debugging.

5. Think Idempotency First

Design transactional operations to handle network retries safely from day one.

6. Document Everything

An undocumented API is broken. Leverage automated specs to keep docs in sync.

7. Version Early

Establish a versioning plan before launching to production, anticipating inevitable future changes.

8. Secure by Default

Enforce HTTPS, validation schemas, token authentication, and rate limiting from day one.

9. Keep It Simple

Don't over-engineer. Rely on standard REST practices before adopting more complex protocols.

10. Evolve Without Breaking

Adhere to strict backward-compatibility rules. Only make additive changes.
